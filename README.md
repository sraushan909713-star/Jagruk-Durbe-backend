## The Problem

In Durbe вҖ” a village in Gaya, Bihar вҖ” broken roads stay broken for years. Drains crack open and nobody fixes them. Panchayat Bhawans sit empty while children have nowhere to read. Government schemes exist on paper but rarely reach the people who qualify. Elected representatives make promises during elections and disappear after.

The gap is not money. The gap is **accountability infrastructure**: villagers have no organized way to document issues, no shared record of what was promised, and no collective pressure mechanism. Complaints stay individual. Demands stay informal. Memory is short.

Jagruk Durbe is the backend that powers an accountability cycle: **Gram Awaaz** (report issues with evidence) вҶ’ **Vikas Prastav** (propose development of existing assets) вҶ’ **Neta Ka Report Card** (rate elected representatives) вҶ’ **Promises** (paper trail of what was promised, witnessed, and whether it was delivered).

**Design principles**

- **Soft delete everywhere.** Every model carries `is_active`, `created_at`, `updated_at`, and `village_id`. Nothing is ever truly deleted вҖ” preserving the audit trail matters for civic accountability.
- **Role-based permissions** at the router layer: `user`, `admin`, `super_admin`. Creators can edit their own content; admins can delete anything.
- **OTP-gated state changes.** Registration, password reset, and account deletion all require OTP verification, with a purpose enum that prevents an OTP issued for one flow from being replayed in another.
- **NSFW filtering at the upload boundary.** Every photo that enters the system is screened by NudeNet before it ever reaches Cloudinary вҖ” protecting villagers from inappropriate content on a public community feed.
- **Signed Cloudinary uploads.** Image uploads route through the backend so the Cloudinary API secret stays server-side. The frontend never holds upload credentials.
- **Built for Durbe.** The app is purpose-built for one village вҖ” Durbe. Models carry `village_id` because that's good schema hygiene, but the wider design (hardcoded content, single-village admin model, Durbe-specific copy) is intentionally single-village. Expansion to other villages, or government adoption, would require a deliberate redesign вҖ” not a config flag. That redesign would be welcome if it serves more people.

## Feature Modules вҖ” 16 Routers, 86 Endpoints

| Module | Purpose |
|---|---|
| **auth** | Phone-based registration, OTP verification, login, password reset, account deletion |
| **gram_awaaz** | Citizens report broken infrastructure with up to 4 photos, geo-tag, and affected count. Community upvotes amplify pressure on authorities |
| **vikas_prastav** | Development proposals for unused village assets (e.g. converting empty Panchayat Bhawans into libraries). Includes estimated cost and funding-source fields |
| **neta_report_card** | Citizens rate elected representatives on multiple dimensions. 6-month cooldown per voter, enforced at the query layer. Aggregated public ratings |
| **promises** | Track promises made by representatives, witnessed by named citizens. Creates an accountability paper trail |
| **schemes** | Government scheme catalog with eligibility, document checklist, and YouTube guides. MyScheme integration planned via in-app WebView in a later phase |
| **contacts** | Directory of officials (Mukhiya, Sarpanch, BDO, MLA) and service providers (doctors, plumbers, carpenters) with phone, designation, and office hours |
| **guides** | Step-by-step documentation guides вҖ” how to get a birth certificate, land record, voter ID, etc. |
| **vendor_listings** | Local vendor crop prices in buy/sell mode. Farmers see daily rates without visiting the mandi; tap-to-call goes through the vendor's verified phone |
| **items** | Master crop/item catalog backing vendor listings (paddy, wheat, mustard, etc.) |
| **units** | Master unit catalog (kg, quintal, bigha, etc.) |
| **job_alerts** | Sarkari Naukri and private opportunities with eligibility broken down into plain language |
| **community_members** | Verified community members section per scheme вҖ” peer-to-peer trust signal |
| **banners** | In-app announcement banners with 11 visual themes, optional YouTube link, and up to 4 linked contacts |
| **uploads** | Hardened photo upload endpoint: MIME-type check вҶ’ size limit вҶ’ NudeNet NSFW screen вҶ’ signed Cloudinary upload |
| **weather** | Rain alerts and forecast via external weather API |

## Security Highlights

- **JWT (HS256)** signed with an environment-loaded 64-byte URL-safe random `SECRET_KEY`
- **bcrypt** password hashing via Passlib вҖ” no plaintext storage anywhere
- **Per-endpoint rate limiting** with SlowAPI (defined in its own `app/core/limiter.py` to keep the import graph acyclic):
  - `/login` вҖ” 5 requests/minute
  - `/send-otp` вҖ” 3 requests / 5 minutes
  - `/register` вҖ” 3 requests / 10 minutes
  - `/reset-password` вҖ” 5 requests/minute
- **OTP purpose enum** вҖ” OTPs are bound to a single flow (registration / password reset / account deletion) and cannot be replayed across flows
- **HTTPBearer** scheme on every protected route; `decode_access_token` resolves the current user before any handler runs
- **Input validation** at the Pydantic schema layer вҖ” 10-digit Indian phone numbers, 8+ character passwords
- **NSFW screening** on every photo upload before it touches public storage
- **Signed Cloudinary uploads** вҖ” secret never leaves the server
- **Strict `.env` discipline** вҖ” no hardcoded defaults for `SECRET_KEY`, `DATABASE_URL`, or Cloudinary credentials. Server fails loud at boot if any of them are missing
- **`.env` is in `.gitignore`** and has never been committed

## Getting Started

### Prerequisites
- Python 3.12+ (project uses 3.14 in development)
- pip, virtualenv

### Setup

```bash
# 1. Clone
git clone https://github.com/sraushan909713-star/Jagruk-Durbe-backend.git
cd Jagruk-Durbe-backend

# 2. Create and activate a virtual environment
python3 -m venv venv
source venv/bin/activate          # macOS / Linux
# venv\Scripts\activate           # Windows

# 3. Install dependencies
pip install -r requirements.txt

# 4. Configure environment
cp .env.example .env
# Edit .env with your own values (see below)

# 5. Run migrations
alembic upgrade head

# 6. (Optional) Seed master data вҖ” Mandi items and units
python -m app.seed.mandi_seed

# 7. Start the server
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
```

Open `http://localhost:8000/docs` for the interactive Swagger UI, or `http://localhost:8000/redoc` for ReDoc.

### Environment Variables

Create a `.env` file in the project root:

```env
DATABASE_URL=sqlite:///./gramseva.db
SECRET_KEY=<generate-with-python-secrets-token_urlsafe-64>

# Cloudinary (signed image uploads)
CLOUDINARY_CLOUD_NAME=<your-cloud-name>
CLOUDINARY_API_KEY=<your-api-key>
CLOUDINARY_API_SECRET=<your-api-secret>
```

Generate a strong `SECRET_KEY`:

```bash
python3 -c "import secrets; print(secrets.token_urlsafe(64))"
```

**Note:** The server intentionally has no default values for these вҖ” it will fail to start if any are missing. This prevents accidentally running production with development secrets.

## Database & Migrations

- **Development DB**: SQLite (`gramseva.db`) вҖ” zero-config, file-based, fast iteration
- **Production target**: PostgreSQL on Railway вҖ” same SQLAlchemy models, driver via `psycopg2-binary`
- **Schema management**: Alembic вҖ” every schema change is a tracked migration file
- **`Base.metadata.create_all()` is intentionally NOT called at startup** вҖ” only Alembic creates or alters tables, so production migrations stay predictable

### Common Alembic Commands

```bash
# Create a new migration after editing models
alembic revision --autogenerate -m "describe the change"

# Apply pending migrations
alembic upgrade head
```

## What I Built and Why

The idea took **years** to mature. Living in Durbe and watching the same broken road, the same empty Panchayat Bhawan, the same unclaimed schemes year after year, I slowly understood that the problem wasn't apathy and it wasn't money вҖ” it was the absence of any structured way for villagers to document, organize, and apply pressure. Once that became clear, the shape of the solution became clear too.

The **build** took six months. I am the sole developer вҖ” backend, frontend, database design, deployment configuration, design system, security pass, and product decisions all done end-to-end. The harder, less visible work lived in the backend:

- Designing 16 cohesive routers that share a consistent auth + soft-delete + rate-limit + role-permission pattern
- A multi-photo upload pipeline that runs MIME and size checks, then NudeNet NSFW screening, then a signed Cloudinary REST upload вҖ” so inappropriate content can't reach the village feed
- An OTP system with a purpose enum so the same code cannot be replayed across registration, password reset, and account deletion flows
- Migrating `photo_url` вҶ’ `photo_url_1/2/3/4` via Alembic without losing data in development
- A 6-month rating cooldown for Neta Report Card enforced at the query layer, with the exact next-allowed date returned in the error response
- A Mandi v2 redesign with master `items` and `units` tables, a `mode=buy/sell` axis on listings, and tap-to-call via a verified phone join вҖ” replacing the looser v1 schema
- Removing `Base.metadata.create_all()` from app startup so Alembic is the single source of truth for schema changes
- Strict env hygiene: no default secrets, fail-loud at boot, `.env` never committed

This isn't a tutorial project. It's a production-shaped system built to serve one specific village in Bihar вҖ” and possibly more, if the work is done right.
